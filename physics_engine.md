## 1. 概览：引擎边界与数据流

```
┌──────────┐        ┌─────────────┐         ┌──────────────┐
│  Sensor  │ a, ω   │ ThrowSys    │ v0,     │ Sim          │ p, v, fwd, up
│ (Device  │ ─────► │ (检测出手)   │ pitch,  │ (积分气动力) │ ──── 落地 ──►
│  Motion) │        │             │ roll    │              │
└──────────┘        └─────────────┘         └──────────────┘
   60 Hz              事件触发                每帧 60 Hz
```

| 模块 | 输入 | 输出 | 主要负责 |
|---|---|---|---|
| `Sensor` (W3C DeviceMotion) | 浏览器事件 | `a` (m/s²), `ω` (rad/s) | 原始数据采集 + 频率监控 |
| `ThrowSys` | `a, ω` 流 | `v0, pitch, roll, fwd, up` | 状态机 (IDLE→ARMED→SWINGING→RELEASED) + 出手参数解算 |
| `Sim` | `v0, fwd, up` | 轨迹、距离、最高点、时长、侧偏 | 三力 (升、阻、重) 积分 |

**关键设计决定**：
- 飞行物理**不用 Cocos 内置刚体**——它解算碰撞但不算升力/阻力/失速，对纸飞机不合适。
- 出手姿态**用加速度计直接解算重力向量**，不用陀螺仪积分（避免 W3C 在不同设备上 alpha/beta/gamma 轴定义不一致的坑）。
- 飞行中机身姿态**不是完全锁死**，引入 `attitudeTau` 做被动对齐，模拟纸飞机重心-气压中心偏置带来的自动俯仰修正。

## 2. 坐标系定义（三套）

整个系统涉及 **3 个右手坐标系**，必须分清楚才能避免符号错误。

### 2.1 世界系 (World, `_w`)

Cocos Creator 默认右手系，**镜头初始朝 -Z**：

```
        +Y (上)
         │
         │
         └────── +X (右，玩家视角)
        ╱
      +Z (背向场景，朝玩家身后)
```

- `+X_w`：玩家右手方向
- `+Y_w`：竖直向上
- `+Z_w`：背向场景，朝玩家
- **飞行前进方向 = -Z_w**

**约定**：本文所有「距离」「侧偏」都按此系测量。

### 2.2 手机本体系 (Phone, `_p`)

W3C `DeviceMotionEvent` 默认手机系，也是右手系：

```
        +Y_p (机顶 / 听筒侧)
         │
         │
         └────── +X_p (机右 / 物理按键侧)
        ╱
      +Z_p (屏幕外 / 出屏方向)
```

- `+X_p`：机右（物理按键侧）
- `+Y_p`：机顶（听筒/前置摄像头侧）
- `+Z_p`：屏幕外（垂直屏幕指向用户）

### 2.3 机身系 (Body, `_b`)

物理仿真用，**不显式写出三个轴**，但隐含：

- `fwd_b`：机头方向（正前），世界系下用变量 `fwd` 表示
- `up_b`：机背方向（正上），世界系下用变量 `up` 表示
- `right_b = fwd_b × up_b`：右翼方向（不显式存，需要时计算）

`fwd` 与 `up` 始终保持**正交且单位长度**，由发射瞬间初始化，飞行中只通过被动对齐改变。

## 3. 状态变量字典

### 3.1 `Sim` 类持有的状态

| 变量 | 类型 | 单位 | 含义 |
|---|---|---|---|
| `p` | Vec3 | m | 飞机在世界系下的位置。初始 `(0, startAlt, 0)`，落地条件 `p.y ≤ 0` |
| `v` | Vec3 | m/s | 世界系下的速度向量 |
| `fwd` | Vec3 (单位) | — | 机身**前向轴**（机头方向）在世界系下的方向 |
| `up` | Vec3 (单位) | — | 机身**上方轴**（机背方向）在世界系下的方向 |
| `t` | scalar | s | 累计仿真时间 |
| `maxAlt` | scalar | m | 飞行中遇到的最大 `p.y` |
| `alpha` | scalar | rad | 当前迎角（用于 HUD 显示） |
| `speed` | scalar | m/s | `\|v\|` 缓存 |

### 3.2 落地结算的派生量

| 字段 | 计算 | 物理含义 |
|---|---|---|
| `distance` | `-p.z` | 沿基准线（世界 -Z）的飞行距离。注意 forward = -Z，所以飞行中 `p.z` 单调减小 |
| `lateral` | `p.x` | 横向偏移（带符号）。`+` 为向玩家右侧偏 |
| `maxAlt` | `max(p.y)` | 最高点高度 |
| `time` | `t` 累计 | 总飞行时长 |

> **MRD §4 的「距离」语义**：飞机落地点到基准线作垂线，读垂足在标尺上的刻度。基准线是从原点指向 -Z 的射线，所以垂线长度即 `|p.z|`，与 `lateral` 解耦。

## 4. 姿态：pitch / roll / 迎角的精确定义

这一节最容易翻车，所以把每个角的**符号约定**和**几何含义**都写死。

### 4.1 Pitch（俯仰角）

> 机头相对水平面的抬头/低头角。**握姿空间**和**飞行空间**两套语义需要分开理解。

**握姿空间（出手前）**：相对于「标准握姿」（机底水平指前方），机底**抬高** = pitch+，机底压低 = pitch-。
代码实现（`ThrowSys._release`，`_liveAng`）直接用重力在手机系下的投影解：

```js
pitch = atan2(-a.y, -a.x)   // 标准握姿下 a_phone ≈ (-g, 0, 0)
```

推导：标准握姿下 `g_phone = (g, 0, 0)` 因为重力指向手机 +X（机右）。绕机身 right 轴（手机 -Z）旋转 +θ 后，重力变为 `(g·cos θ, g·sin θ, 0)`，所以手机 a 读数变为 `(-g·cos θ, -g·sin θ, 0)`，反算 `θ = atan2(-a.y, -a.x)`。

**飞行空间（已发射）**：飞行中**不再追踪 pitch 标量**，而是直接维护 `fwd` 单位向量。需要时 pitch 可以从 `fwd` 反算 `pitch = asin(fwd.y)`。

**范围**：合法范围 `±60°`（超出 `_release` 拒绝并要求重投）。

### 4.2 Roll（横滚角）

> 机身绕机头方向的旋转。`+` 为右倾（右翼下沉），`-` 为左倾。

**握姿空间**：

```js
roll = atan2(a.z, -a.x)   // 标准握姿下 a_phone ≈ (-g, 0, 0)
```

推导：绕机头（手机 -Y）旋转 +φ 后，`g_phone = (g·cos φ, 0, -g·sin φ)`，所以 a 读数 `(-g·cos φ, 0, g·sin φ)`，反算 `φ = atan2(a.z, -a.x)`。

**飞行空间**：roll 不再单独追踪，体现在 `up` 向量与世界 `+Y` 的夹角中。

### 4.3 迎角 α (Angle of Attack)

> **速度方向**与**机身前向**之间的夹角，**带符号**。是升力/阻力计算的核心输入。

```
        body up
          │
          │   ↗ v (速度，斜下)
          │  ╱
          │╱  ← α 在这里 (>0)
fwd ──────┴───────►
```

约定（`Sim._sub`）：

- **α > 0**：速度方向**在机身下方**（机头相对气流抬起）→ 正常飞行最常见
- **α < 0**：速度方向**在机身上方**（机头相对气流低头）→ 飞机被动俯冲

代码实现：

```js
cosA = v_hat · fwd                          // 余弦 = 内积
α    = acos(cosA)                           // 始终 ∈ [0, π]
lat  = v_hat - fwd · cosA                   // v_hat 中垂直于 fwd 的分量
if (lat · up > 0)  α = -α                   // 速度在 body up 一侧 → 负迎角
```

**符号检查**：发射瞬间 `v` 沿 `fwd`（α=0），重力把 `v` 拉向下。「下」意味着 `lat · up < 0`（因为 up 指机背），所以 α 保持正号 → 升力 +。这是设计正确的。

## 5. 坐标变换

### 5.1 手机系 → 世界系

MRD §3.1 规定的握姿四点（前三点独立，第四点由叉乘唯一确定）：

| 世界方向 | 手机本体轴 |
|---|---|
| 前 (-Z_w) | -Y_p (机底) |
| 上 (+Y_w) | -X_p (机左) |
| 右 (+X_w) | -Z_p (屏幕背向) |
| 左 (-X_w) | +Z_p (屏幕外) ← 第四点，由前三点叉乘推出 |

由此得到从手机系到世界系的常值旋转矩阵 R（列为各手机轴在世界系下的方向）：

```
R = [[ 0, 0, -1],     ← 世界 +X = 手机 -Z
     [-1, 0,  0],     ← 世界 +Y = 手机 -X
     [ 0, 1,  0]]     ← 世界 +Z = 手机 +Y
```

写成代码（`prototype.html` L166）：

```js
phoneToWorld(v) = (-v.z, -v.x, v.y)
```

对应的等价四元数（`Q_CANON`）：

```js
Q_CANON = (0.5, -0.5, -0.5, 0.5)   // (x, y, z, w)
```

可验证：`Q_CANON · (1, 0, 0)_p = (0, -1, 0)_w`，即手机 +X (机右) 转到世界 -Y (向下) ✓

### 5.2 重力在手机系下的投影

握姿验证用：世界重力 `g_w = (0, -g, 0)`，反变换到手机系（`R^T`）：

```
g_p = R^T · g_w = (R^T · (0,-g,0))
     = (-g·R[1,0], -g·R[1,1], -g·R[1,2])    ← R[1,*] 是世界 +Y 行
     = (-g · (-1), -g · 0, -g · 0)
     = (g, 0, 0)
```

所以**标准握姿下手机系重力是 `(g, 0, 0)`**（指向机右，机右朝下），加速度计读数（W3C 比力约定，等于 -gravity）应为 `(-g, 0, 0)`。这是 `poseMain`/`poseTol` 检查的物理依据。

### 5.3 出手姿态 → 世界系 fwd/up

`ThrowSys._release` 给出 `pitch`、`roll`，需要重建世界系下的 `fwd`、`up`。两步法：

**Step 1**：先 pitch（绕世界 +X 旋转 p）：

```
fwd = (0, sin p, -cos p)         ← 抬头时 +Y 分量为正
up_base = (0, cos p, sin p)
right = fwd × up_base = (1, 0, 0)
```

**Step 2**：再 roll（绕新 fwd 轴旋转 r）：

```
up = up_base · cos r + right · sin r
   = (sin r, cos r · cos p, cos r · sin p)
```

**符号验证**：roll = +30°（右倾）→ `up.x = sin 30° = 0.5`，up 向量指向 `+X` 一侧（玩家右侧），即机背向右倾，机腹向左 → 升力水平分量推飞机向右。✓

代码：`prototype.html` L399–402（ThrowSys 内）和 L793–797（btnTest 测试发射）实现这两步。

## 6. 发射瞬间：从握姿到机身姿态

完整链条（`ThrowSys` 内部）：

```
1. ARMED 阶段（200 ms 静止窗口）
   └─ 收集 a 样本，求 a_rest 均值
   └─ 校验：a_rest.x < -0.85g 且 |a_rest.y|, |a_rest.z| < 0.4g
   └─ 校准成功后强制 q := Q_CANON （此后陀螺仪积分不影响初始姿态解算）

2. SWINGING 阶段
   └─ 每帧用 ω 积分四元数 q（仅供 a_lin_world 反推用，不进入 release）
   └─ a_lin_phone = a + g_phone   (加速度计 = 比力 = a_lin - g)
   └─ a_lin_world = q · a_lin_phone
   └─ a_fwd = -a_lin_world.z      （世界 forward = -Z）
   └─ 维护 800 ms 滚动窗口

3. RELEASED 触发
   └─ 检测 a_fwd 三样本局部峰值 > triggerAccel
   └─ v0 = ∫ a_fwd dt (从 a_fwd<0.5 的 startIdx 到 peak) × kImpulse
   └─ pitch、roll 从挥动开始前 5 帧的 a 平均值反解（重力向量法）
   └─ 用 §5.3 重建 fwd, up，传给 Sim.launch
```

**关键设计**：pitch/roll **不用陀螺仪积分**（积分漂移 + iOS Safari 轴定义混乱），用挥动起始前的稳定重力向量直接反解。这一选择在 §3.2 of `tech_design.md` 有详细论证。

## 7. 飞行中：每一步在做什么

`Sim._sub(dt)` 是物理仿真的最小单元，每帧调用 `subs` 次。

### 7.1 三个外力

合力 `F = F_lift + F_drag + F_gravity`，加速度 `a = F / mass`。

**重力**：恒定 `F_g = (0, -m·g, 0)`。

**升力**（`Sim._sub` L486）：

```
|F_lift| = ½ · ρ · |v|² · S · CL(α)
```

- `ρ`：空气密度（`rho` 参数，默认 1.225 kg/m³）
- `|v|²`：速度平方 → 速度翻倍升力变 4 倍，是 phugoid 振荡的能量来源
- `S`：翼面积（`area` 参数，默认 0.02 m² ≈ 半张 A4）
- `CL(α)`：升力系数函数（详见 §7.2）

升力**方向**：垂直于 v，且位于 (v, body_up) 张成的平面内。算法：

```js
inner   = body_up × v_hat        // 与两者都垂直 → "横向"轴
liftDir = v_hat × inner          // 在 (v, up) 平面内、与 v 垂直
```

直观理解：飞机在空中，升力沿机身上方推，但被速度方向"拉斜"——速度有多斜（pitch up/down），升力方向就斜多少。**这是为什么横滚后会侧偏**：横滚把 `body_up` 倾斜，升力方向跟着倾斜，水平分量推飞机侧向移动。

**阻力**（`Sim._sub` L487）：

```
F_drag = -½ · ρ · |v|² · S · CD · v_hat
```

方向永远与速度反向。`CD` 计算见 §7.3。

### 7.2 升力系数 CL(α) — 当前实现

分三段，由 `alphaStall`（默认 12°）控制：

| 区段 | 范围 (deg) | 公式 | 物理意义 |
|---|---|---|---|
| 线性段 | \|α\| ≤ alphaStall | `CL = 2π · α` (rad) | 薄翼理论极限 |
| 失速过渡 | alphaStall < \|α\| ≤ alphaStall+6° | 从峰值线性降到 `CLpostStall` | 翼面气流分离 |
| 深失速 | \|α\| > alphaStall+6° | `CL = sign(α) · CLpostStall` | 完全分离，残余升力 |

**关键观察**：`CL_max ≈ 2π · alphaStall(rad)`，所以 `alphaStall` 同时影响失速角和峰值升力。α=12° 时 CL≈1.32。

### 7.3 阻力系数 CD

```
CD = CD0 + k · CL²
```

- `CD0` (默认 0.06)：零升阻力——表面摩擦 + 形阻
  - **软下限钳位**：当 CD0 < 0.03 时用二次过渡到 0.03（防止 UI 滑块拉到极小值时飞太远，已修复）
- `k · CL²` (k默认0.10)：诱导阻力——产生升力本身就要付出的代价
  - k ≈ 1/(π·AR·e)，纸飞机展弦比 AR≈3.5、效率 e≈0.9

升阻比 L/D = CL/CD 决定滑翔效率。

### 7.4 积分（semi-implicit Euler）

```
F = F_lift + F_drag + F_gravity
a = F / mass
v += a · dt        // 先更新速度
p += v · dt        // 再用新 v 更新位置
```

semi-implicit Euler 比显式 Euler 更稳，对周期运动（phugoid）能量误差小得多。

## 8. 被动机身对齐 (attitudeTau)

### 8.1 为什么需要

如果机身姿态完全锁死（fwd 永不变），飞机进入「势能 ↔ 动能」不衰减循环：

```
下落 → v 偏向下 → α 增大 → CL 飙升 → 升力远超重力 → 弹回去
弹回去 → 速度损失 → α 反向 → 又掉下来 → 循环
```

这是经典 **phugoid 长周期振荡**。理论上对真飞机也存在，但纸飞机的振荡会很快被一种「自动俯仰修正」机制压住——**重心稍位于气压中心之前**，所以当 α 变大时机身被气流推得低头，α 变小时被推得抬头，自动对齐速度方向。

### 8.2 实现：把 fwd 向 v_hat 松弛

不建复杂的力矩模型，直接在每个子步把姿态向速度方向拉近一点：

```js
ang   = arccos(fwd · v_hat)               // 当前夹角
blend = min(1, sdt / attitudeTau)         // 本步对齐多少比例
绕 (fwd × v_hat) 轴旋转 (ang · blend)
应用到 fwd 和 up（保证 up 也跟着转，正交关系不破）
```

- `attitudeTau` 是**时间常数**（s）：经过 τ 时间，fwd 与 v_hat 之间剩余的夹角约为初始的 1/e ≈ 37%
- τ 越小：对齐越快，α 几乎贴着 0，升力刚好 ≈ 重力，phugoid 几乎被压平
- τ 越大：对齐越慢，phugoid 越明显
- τ ≥ 5：实质等于关闭，回到固定姿态行为

**副作用**：被动对齐**也会同时压住起飞抬头的有效 α**——你给一个 pitch=20° 的发射，但 v 一开始水平（沿 fwd），所以 α=0。飞着飞着 v 在重力作用下开始下垂，本来 α 该增大产生升力托住飞机，但 attitudeTau 把 fwd 拉去追 v → α 始终接近 0 → 升力不足 → 飞机下落得比预期更快。

**实践**：τ ∈ [0.3, 0.6] 体感最好——既能压住明显波浪，又留有一点延迟让升力起作用。

## 9. 积分稳定性与动态子步

### 9.1 稳定性条件（理论）

semi-implicit Euler 的能量误差与 `(a · dt)²` 同阶。当 L/W 极大时（如 area=0.02, mass=0.001 → L/W≈133），单步 `Δv = a · dt` 能与速度本身同量级，导致**符号振荡**：

```
frame 0:  v_y=+1.87 (升力托起)
frame 1:  v_y=+1.79 → α = -0.84° (升力变负)
frame 2:  v_y=+2.16 → α = -5.03° (剧烈变化)
frame 3:  v_y=+3.11 → α = -12.35° (失速边缘)
... 数值发散 → maxAlt=969m, dist=685万米
```

诊断标准：`Δv / v > 0.5` 时积分发散。

### 9.2 动态子步实现（已修复）

`Sim.step` 顶部估算「最坏情况单步速度变化」并据此计算所需子步数：

```js
qd_est    = 0.5 · ρ · v²
FL_max    = qd_est · S · (2π · αstall)        // 失速角处的最大升力
net_accel = max(0, FL_max / mass - g)         // 净垂直加速度
subsAccel = ceil(net_accel · dt / (0.3 · v))  // 要求 Δv/v < 0.3
subsBase  = min(maxSubsteps, ceil(dt / (1/120)))  // 原有逻辑
subs      = min(64, max(subsBase, subsAccel)) // 仅当极端时扩展，上限 64
```

**关键设计**：用 `max(subsBase, subsAccel)` 而非 `max(maxSubsteps, ...)`，确保正常参数下子步数与原版完全一致（回归测试 0% 偏差）。

## 12. 飞行现象解读：roll 为什么"过顶后才转得明显"

### 12.1 现象

3D 视窗 + 时间轴回放观察：发射 `v0=9, pitch=12°, roll=15°`，俯视图看是一条**先直后弯**的钩形——爬升段几乎走直线，过最高点后侧偏迅速增大。实测数据（`_verify_roll.py` 跑当前模型）：

| 阶段 | 用时 | pos.x 增量 |
|---|---|---|
| 起飞 → 顶点 (apex) | 0.83 s | **0.61 m** |
| 顶点 → 落地 | 2.27 s | **5.76 m** |
| 总计 | 3.10 s | 6.37 m |

下降段的横向位移**是上升段的 9 倍**。直觉上"过顶后转弯加剧"是没错的，但**根本不是因为 a_lat 在过顶后突然变大**。

### 12.2 横向力的分量

```
F_lat_x = ½·ρ·v² · S · CL(α) · liftDir.x      （升力贡献，CL 带号）
        - v_hat.x · ½·ρ·v² · S · CD            （阻力贡献，永远反向）
a_lat   = F_lat_x / mass
```

三个时变因子：① v²(t)，② CL(α(t))，③ liftDir.x(t)。

### 12.3 实测时间序列（每 0.1 s 采样）

```
   t   pos.x   vel.x   a_lat    α°      CL    liftDirX
 0.02   0.000   0.023    +2.5   0.5°   0.05    0.26
 0.10   0.011   0.249    -6.8  -1.6°  -0.18    0.26     ┐
 0.20   0.042   0.472   -10.2  -3.2°  -0.35    0.26     │ attitudeTau
 0.30   0.091   0.502   +13.6  +5.4°  +0.59    0.26     │ 高频振荡
 0.40   0.160   0.829   -10.7  -5.8°  -0.63    0.25     │ α 在 ±10° 间
 0.50   0.244   0.936    -9.5  -6.5°  -0.71    0.26     │ 来回切换
 0.60   0.341   1.037    +9.2  +7.9°  +0.87    0.26     ┘
 0.83◆  0.614   1.313    +8.0 +11.2°  +1.23    0.28      ◆ 顶点
 1.00   0.840   1.438    +4.6 +15.0°  +0.96    0.30
 1.50   1.764   2.251    -3.9 -16.1°  -0.82    0.29     ┐ α 振幅减小，
 2.00   2.975   2.497    +8.0 +10.7°  +1.18    0.38     │ liftDirX 上涨
 2.50   4.336   2.980    +6.5  +5.8°  +0.63    0.40     │
 3.00   5.998   3.647    +4.7  +2.7°  +0.29    0.41     ┘
```

**两个关键观察**（与朴素直觉相反）：

1. **a_lat 的瞬时值在爬升期高频振荡，正负交替**——这是 attitudeTau=0.3s 的纵向阻尼模态。fwd 追 v_hat 时反复过冲、回拉，CL 跟着 α 一起振荡。**时间平均**才是正的。

2. **a_lat 没有"过顶后突然变大"**——平均量级整个飞行都在 5 m/s² 上下。后期 liftDir.x 从 0.26 缓慢涨到 0.41（+58%），是个温和的趋势，不是阶跃。

### 12.4 那钩形是怎么来的？答案在「双重积分」

`pos.x = ∫∫ a_lat dt²`。**a_lat 平均接近常数时，pos.x 仍然按 t² 增长**——后半段位移天然比前半段大得多。

把整局拆成爬升 / 下降两段后看：

| 量 | 爬升段平均 | 下降段平均 | 比值 |
|---|---|---|---|
| 时长 Δt | 0.83 s | 2.27 s | **2.7×** |
| vel.x 平均 | ~0.7 m/s | ~2.5 m/s | **3.5×** |
| pos.x 增量 = vel.x · Δt | 0.61 m | 5.76 m | **9.5×** |

下降段的位移占绝对优势是这两个因子相乘的结果：**时间长 2.7 倍 × 横向速度大 3.5 倍 ≈ 9 倍位移差**。

### 12.5 为什么下降时间是上升的 2.7 倍？

纯几何 + 重力做功：

- 起飞高度 = 1.5 m，最高点 = 2.13 m → **上升只爬了 0.63 m**
- 顶点到地面 = 2.13 m → **下降要跌 2.13 m**
- 高度比就 3.4×，加上下降阶段升力分量减速，时间差 ~2.7×

如果把 `startAlt` 调成 0，让飞机从地面出发，上升和下降时间会接近，钩形会立刻**对称化**（变成一个均匀曲弧）。这是该现象**纯由发射条件决定**的最直接证据。

### 12.6 高频 α 振荡的根源

trace 里 α 在 ±10° 之间快速翻号（周期 ~0.1–0.2 s），这是 **attitudeTau 与 CL 的负反馈环**：

```
v 偏离 fwd → α 增大 → CL 增大 → 升力把 v 推回去 → α 减小 → 越过零点继续 → α 反向
   ↑                                                                            │
   └────────────────────── attitudeTau 同时把 fwd 拉向 v ──────────────────────┘
```

`attitudeTau=0.3` 这个值刚好让该回路欠阻尼（damping ratio ~0.4），所以振铃明显。**这跟 phugoid 不是同一个模态**——phugoid 周期 5–10 s，由能量交换驱动；这里的振荡周期 0.1 s，由**短周期纵向稳定模态**驱动，是 `attitudeTau` 太大造成的（真实纸飞机的对应时间常数约 0.05 s）。

### 12.7 平板模型（优化 A）会不会改变钩形？

会变浅，**但不会变平**：

| 模型 | dist | lateral | 钩形比例 (lateral/dist) |
|---|---|---|---|
| 当前 `2π·α` | 5.85 m | 6.40 m | 1.09 |
| 平板 `sin(2α)` | 6.68 m | 5.18 m | 0.78 |

平板模型把 CL_max 砍到 1/3，所以瞬时 a_lat 量级也降到 ~1/3，钩形自然浅一些。但**起飞位移 / 落地位移的比例（10:1）仍然类似**——因为 §12.5 的几何不对称没变。

### 12.8 两条让转弯更"线性"的路径

如果产品想让横滚响应更可预测（例如玩家觉得"我倾 15° 期望弯一倍"），改 CL 模型治标不治本：

1. **改发射条件**：让飞机从地面附近平抛（`startAlt ≈ 0`，`pitch ≈ 0`），上升下降对称。最直接，不动物理。
2. **改 attitudeTau**：调到 0.1 s 以下，让 fwd 紧跟 v_hat → α 始终接近 0 → CL 接近 0 → 升力主要靠 roll 制造的水平分量。横向加速度时间历程会接近常数，钩形变均匀曲线。代价：phugoid 抑制更强，"飘"的感觉消失。
3. **改 lift 的方向算法**：当前 `liftDir = perp(up, v_hat)` 在 α=0 时给出 body up。如果把"主升力"和"横滚水平分量"显式拆开，可以让横滚效应解耦于 α 振荡——但这是优化 C（侧滑模型）的范畴。

简言之：**钩形不是 bug，是当前模型 + 发射条件几何共同决定的产物**。要不要改它取决于产品意图。

## 13. 侧滑现象：fwd 跟 v_hat 在水平方向不齐

### 13.1 现象

在 3D 视窗里发射 `roll=15°` 投掷，拖时间轴到飞行中段（t≈1.0–2.0 s），切到俯视图，可以看到：
**黄色速度向量已经明显往 +X 转**（向"内圈"），**但青色机头方向 fwd 还落在更靠后的方位上**——飞机看上去在斜着"侧滑"。

真实纸飞机和真飞机几乎不会有这种持续侧滑：垂直尾翼的"风向标"效应让机头紧跟速度。所以这个体感不真实。

### 13.2 量化：稳态偏角 ≈ 转弯率 × 时间常数

§8.2 的 attitudeTau 是一阶松弛。把它写成微分方程的形式（fwd 在 v_hat 方向上的角度差为 Δθ）：

```
d(Δθ)/dt = -Δθ / τ + Ω
```

其中 Ω 是 v_hat 自身的转动角速率（fwd 要追的"移动靶")。**稳态解** `d(Δθ)/dt = 0` 给出：

```
Δθ_steady = Ω · τ
```

实测验证（v0=9, pitch=12°, roll=15°，τ=0.3）：

```
   t (s)    sp     v_yaw°  fwd_yaw°   lag°
   0.10    7.76    +1.9     +0.2       1.6     ← 起步
   0.50    4.59   +11.9     +5.4       6.5
   1.00    2.53   +34.8    +21.8      13.0     ┐
   1.50    2.57   +64.1    +48.1      16.1     │ 稳态 ~12°，
   2.00    2.72   +80.8    +70.2      10.5     │ 瞬态峰 16°
   2.50    3.25   +89.7    +83.9       5.8     ┘
   3.00    4.01   +93.7    +90.9       2.8     ← 转弯结束，lag 收敛
```

中段 (1–2.5 s) 平均 lag = **12.3°**，对应：

- v_hat 实测水平转弯率 Ω = 36 °/s ≈ 0.63 rad/s
- 公式预测 Δθ = 0.63 × 0.3 = 0.19 rad ≈ **11°** ✓

视觉上，在整个转弯主段，**机头都比速度方向落后 10–16°**。

### 13.3 真实纸飞机的"风向标"机制

真实纸飞机有三根**独立**的姿态稳定轴，各自时间常数不同：

| 轴 | 物理来源 | 恢复矩 ∝ | 实测 τ |
|---|---|---|---|
| pitch | 重心-气压中心偏置 + 水平尾翼 | α (迎角) | ~0.05–0.1 s |
| **yaw** | **垂直尾翼（weathercock）** | **β（侧滑角）** | **~0.05 s** |
| roll | 翼面横滚阻尼 | p (滚转率) | ~0.1 s |

如果 τ_yaw = 0.05s，同样 Ω=0.63rad/s 时，Δθ_steady ≈ 0.03 rad ≈ **2°**。视觉上根本看不出。

### 13.4 为什么当前模型 pitch 还行，yaw 不行

代码里 (`prototype.html` `Sim._sub` L605-616) 用**单一 τ=0.3s 的旋转**追 v_hat，pitch 和 yaw **共用同一个时间常数**。但两个方向的体感差异极大：

| 轴 | 是否有平衡点 | 实际 lag | 视觉效果 |
|---|---|---|---|
| pitch | **有**：升力 ≈ 重力 时 α 自然收敛到稳定小值 | 平均 ~5° | 几乎无感 |
| yaw | **没有**：roll 不变 → 横向力持续推 v 偏离 fwd | 持续 10–16° | 明显侧滑 |

pitch 的 lag 之所以小，是因为系统**自发收敛**到 `α ≈ α_trim`（升重平衡点）。yaw 没有这种平衡点——只要 roll 还在，横向力就一直把 v 往侧面推，偏角直接由 `Ω·τ` 决定。

### 13.5 与 §12.6 高频振荡的关系

同一个 attitudeTau 机制的两面表现：

- **§12.6 振铃**（α 在 ±10° 间反复翻号）= fwd 追 v_hat 时**过冲** → 短周期模态欠阻尼
- **§13 侧滑**（fwd 比 v_hat 落后 12°）= fwd 追 v_hat 时**持续滞后** → 稳态相位差

减小 τ 同时缓解两者，但根因都是「pitch / yaw / roll 三轴共用一个 τ」**这件事本身**。

### 13.6 三条修复路径

#### 路径 A · 全局缩小 τ（最快但有副作用）

```js
PARAMS.attitudeTau = 0.1;   // 一行
```

效果：lag 从 12° 降到 4°，肉眼感觉好多了。

代价：phugoid 也被强压平，飞行手感失去"飘"的味道；§12.6 的振铃几乎消失（这部分是好事）。

#### 路径 B · 拆分 τ_yaw 与 τ_pitch（推荐）

把现有 `axRaw = fwd × newVh`（`prototype.html:607`）按世界 +Y 轴分解成两个分量：

```js
const axRaw = V.crs(this.fwd, newVh);
const axLen = V.len(axRaw);
if (axLen > 1e-6) {
  // 拆分：yaw 分量沿世界 +Y，pitch 分量是剩下的
  const yawProj = axRaw.y;
  const yawAxis  = V.c(0, yawProj, 0);
  const pitchAxis = V.sub(axRaw, yawAxis);
  // 各自的角速度（向量长度即旋转角速率方向上的投影）
  const angYaw   = Math.abs(yawProj);                // 已经是 sin·angle 量级
  const angPitch = V.len(pitchAxis);
  const tauYaw = 0.08, tauPitch = 0.3;
  const blendYaw   = Math.min(1, sdt / tauYaw);
  const blendPitch = Math.min(1, sdt / tauPitch);
  // 合成单个旋转向量（小角近似 ok）
  const rotVec = V.add(
    V.scl(yawAxis,   blendYaw   / Math.max(angYaw, 1e-9)   * angYaw),
    V.scl(pitchAxis, blendPitch / Math.max(angPitch, 1e-9) * angPitch)
  );
  // ... 转成四元数应用到 fwd, up（同 §8.2）
}
```

约 15 行，不引入新参数概念。物理意义对应「垂直尾翼比水平尾翼有更强稳定性」。
代入估算：τ_yaw=0.08 → lag 从 12° 降到 ~3°；同时 pitch 的 phugoid 行为完全保留。

#### 路径 C · 显式 weathercock 模型（最物理）

这是 §14.C「侧滑 + 侧向力」优化项的另一半：

- 计算体轴系下侧滑角 `β = atan2(v_lat_body, v_fwd_body)`
- 加入 yaw 恢复力矩 `M_z = -k_β · β`
- 对绕世界 Y 的角动量积分（需要引入转动惯量 `I_yy`）

约 30 行 + 1 个新参数 `I_yy`。优点：与真实飞机机理一致，自然产生 Dutch roll、spiral 等模态；缺点：调参面变大，对一个 30 秒的纸飞机游戏体验提升有限。

## 14. 优化方向（按 ROI 排序）

> ROI = 真实感提升 / 实现复杂度。仅推荐 ★★ 及以上的项落地。

### 优化 A · ★★★ 升力曲线改用平板模型（影响最大，改动小）

**现状代码**（`Sim._CL`）：

```js
if (aAbs <= alphaStall) return 2 * Math.PI * alpha;
```

**问题**：`2π·α` 是「薄翼理论」结果，对**有曲率的翼型**才准确。纸飞机就是一张平板，没有曲率。
实测对比表显示当前模型在小 α 段**高估 CL 约 3 倍**：

| α (°) | 当前 `2π·α` | Hoerner 平板 `sin 2α` | AR=3.5 修正 | 实际纸飞机风洞值 |
|---:|---:|---:|---:|---:|
| 5 | 0.55 | 0.17 | 0.33 | 0.2–0.3 |
| 10 | **1.10** | 0.34 | 0.61 | 0.4–0.7 |
| 12 | **1.32** ← 当前 CL_max | 0.41 | 0.71 | 0.5–0.8 |
| 20 | 0.60 (失速) | 0.64 | 1.04 | 0.7–1.1 |
| 45 | 0.60 (深失速) | **1.00 ← max** | 1.34 | 0.9–1.2 |

**当前模型问题**：在小 α (典型 10–15°) 高估，在大 α 严重低估。失速峰过尖锐，落差太大。

**推荐公式**（Hoerner 平板，统一形式不需要分段）：

```js
_CL(alpha) {
  // 平板纯 Hoerner：CL = sin(2α)，CL_max = 1 出现在 α=45°
  // 物理来源：流体在板的两侧产生压差，无前缘吸力补偿（与曲面翼不同）
  return Math.sin(2 * alpha);
}
```

或带 AR=3.5 修正（升力略增，更接近真实纸飞机）：

```js
_CL(alpha) {
  const AR = 3.5;
  const a0 = 2 * Math.PI / (1 + 2/AR);          // ≈ 4.0
  // 用 sin·cos 而非 α 直接，使大 α 自动过渡
  // 分母项压制 α=45° 附近过冲，CL_max 落在 1.3 左右
  const sa = Math.sin(alpha), ca = Math.cos(alpha);
  return a0 * sa * ca / (1 + 0.7 * Math.abs(sa));
}
```

**Python 实测整局飞行差异**（默认参数）：

| 场景 | 当前 dist/maxAlt | 平板 (sin 2α) dist/maxAlt | 变化 |
|---|---|---|---|
| 水平 (v0=9, p=0) | 11.3 m / 1.5 m | 8.1 m / 1.5 m | -28% / 0% |
| 微抬头 (v0=9, p=12) | 12.7 m / 2.3 m | 9.6 m / 2.0 m | -24% / -13% |
| 大抬头 (v0=9, p=30) | 10.2 m / 3.3 m | 8.7 m / 3.0 m | -15% / -9% |
| 狠抛 (v0=15, p=15) | 16.9 m / 3.3 m | 13.7 m / 2.9 m | -19% / -12% |
| 微滚 (v0=9, p=12, r=10) lat | +6.2 m | +3.7 m | -40% |

**解读**：现在的飞行普遍**飞太远 + 横滚过敏感**，平板模型把这两个偏差同时修正。
实际纸飞机在普通投掷下 5–15 m 是合理范围，当前模型偏向 12 m 上限太多。

**额外好处**：失速无突变，参数 `alphaStall`、`CLpostStall` 可以删除，参数面板更简洁。

**复杂度**：改 1 个函数（3 行），删 2 个参数。**强烈推荐**。

### 优化 B · ★★★ 阻力补 `sin²α` 主导项

**现状**：`CD = CD0 + k · CL²`，纯抛物线极曲线。

**问题**：α 大时（> 15°）平板的阻力**主要来自前后压差**（"开车窗"那种钝体阻力），而非诱导阻力。
`k·CL²` 在 α=30° 时只占总阻力的 1/3，剩下 2/3 来自压差，必须建模。

**修复**：

```js
_CD(CL, alpha) {
  const sa = Math.sin(alpha);
  return CD0_eff + this.P.k * CL * CL + 1.0 * sa * sa;  // ← 新增 sin²α 项
}
```

**效果（实测）**：在大 pitch 投掷下飞机更"沉"：

| pitch | 当前 dist | 平板 (含 CD_sin²α) dist | 变化 |
|---|---|---|---|
| 30° | 10.16 m | 8.71 m | -14% |
| 45° | (推算 ~9 m) | (推算 ~7 m) | -22% |
| 60° | 7.48 m | 5.71 m | -24% |

正是 MRD §3.2 要求的「失速俯冲」效果——大角度抛出去飞机会真的"吃不住升力"。

**复杂度**：1 行代码。**与优化 A 配套使用**。

### 优化 C · ★★ 引入侧滑角 β 与侧向力

**现状**：升力只有 `lift_dir = (v_hat, up) 平面内 ⊥ v`，没有侧向（机身右翼方向）的力。

**问题**：当机身有横滚 + 飞行方向有水平偏差时，真实飞机会产生**侧向力 (Y)**，机翼会有阻尼性的「自动对齐」效应。当前模型用 attitudeTau 是粗糙近似。

**改进**：引入完整的体轴系（fwd, up, right），把速度向量分解到三轴：

```js
right = cross(fwd, up)
v_fwd  = dot(v, fwd)        // 纵向分量
v_normal = dot(v, up)       // 法向分量（α 在此面内）
v_lat  = dot(v, right)      // 横向分量（β 在此面内）
α = atan2(-v_normal, v_fwd)
β = atan2(v_lat, v_fwd)

CY = -k_β · β               // 侧向力系数，反向于 β
F_Y = qd · S · CY · right
```

**效果**：roll>0 飞行时，侧向风会被自然消化，飞机不会无限制侧偏。同时可以**把 attitudeTau 的对齐效应分解为 (α 修正 + β 修正)**，更可控。

**复杂度**：~30 行代码。**对核心玩法影响中等**——主要让 roll 大角度的轨迹更自然。

### 优化 D · ★ 用真正的转动动力学替代 attitudeTau

**当前**：`attitudeTau` 是**纯运动学松弛**——直接旋转 fwd 向量。

**真实**：飞机姿态是**力矩**导致**角加速度**积分出来的：

```
M_pitch = q_dyn · S · c · Cm(α, q_pitch)   // q_pitch = 俯仰角速度
α_ddot = M_pitch / I_yy
ω_pitch += α_ddot · dt
q_body = integrate(q_body, ω_body, dt)
```

需要新增：转动惯量 `I_yy/I_xx/I_zz`、力矩系数 `Cm/Cl/Cn`、力矩臂 `c, b`。

**优点**：
- 完全模拟真实飞机的纵向稳定性边界（CG 位置 → 飞行风格）
- 自然出现 phugoid 阻尼、Dutch roll、spiral 等模态
- 阵风响应正确

**缺点**：
- 复杂度高（需要 ~150 行代码 + 一组新参数）
- 调参难度激增（CG 位置、I_yy、Cm_α、Cm_q ...）
- 对 30 秒、~10m 距离的纸飞机游戏体验改善有限

**建议**：除非要做"飞机模拟器"档次，否则保留 attitudeTau。

### 优化 E · ★ 风场与湍流

**当前**：完全无风。

**简单实现**：

```js
// 静态风（每局随机一次）
wind = (random() - 0.5) * 2  // m/s
v_rel = v - wind_world       // 用相对风速算 α 和 q_dyn
```

**Dryden 湍流模型**（高级）：在 v_rel 上叠加一个 Ornstein-Uhlenbeck 随机过程。

**游戏价值**：增加每局的随机性，但与"真实物理"的关联弱（用户分不出"风"和"投歪了"）。**仅在需要"减少完美投掷"时考虑**。

**复杂度**：~10 行（静态风），~50 行（湍流模型）。

### 优化 F · — RK4 积分器（不推荐）

semi-implicit Euler 已经足够稳定（dt=1/240 = 4 子步/帧）。RK4 计算量 4×、误差降 O(h³)，但当前误差已被 attitudeTau 主导，换 RK4 没有可见改善。**不推荐**。

## 15. 推荐落地路径

### 阶段 1（强烈推荐）：A + B
共约 4 行代码改动 + 删 2 个参数。

```js
_CL(alpha) {
  const sa = Math.sin(alpha), ca = Math.cos(alpha);
  return 4.0 * sa * ca / (1 + 0.7 * Math.abs(sa));   // AR=3.5 修正
}

_CD(CL, alpha) {
  const sa = Math.sin(alpha);
  const CD0_min = 0.03;
  const CD0_eff = this.P.CD0 < CD0_min
    ? CD0_min - (CD0_min - this.P.CD0)**2 / CD0_min : this.P.CD0;
  return CD0_eff + this.P.k * CL * CL + 1.0 * sa * sa;
}
```

PARAMS 中删除 `alphaStall` 和 `CLpostStall`，UI 滑块同步删除。

**预期收益**：
- 普通投掷距离从 12 m 降到 9 m，更接近真实纸飞机（5–15 m 区间）
- 横滚侧偏减弱 ~40%，玩家更容易控制方向

## 16. 投掷状态机与检测逻辑详解

> §6 给出了"发射瞬间"的高层链条，本节把 `ThrowSys`（`prototype.html` L300–470）的状态机、握姿合法性、出手识别、初始状态解算逐条拆开，并指出当前实现里**站不住脚 / 边界模糊 / 与 MRD 字面要求不符**的地方。

### 16.1 状态机总览

四个显式状态（`FLIGHT` 是 UI 层的别名，`Sim` 接手后才置位）：

| 状态 | 进入条件 | 在做什么 | 退出条件 → 下一态 |
|---|---|---|---|
| `IDLE` | 上电 / `reset()` | 等待 UI 触发 `arm()` | 用户按"准备" → `ARMED` |
| `ARMED` | `arm()` 调用 | 收 200 ms 静止样本，验证握姿 | 三轴阈值通过 → `SWINGING`；超时未通过 → 滚动重试（仍在 `ARMED`） |
| `SWINGING` | 握姿校准成功，`q := Q_CANON` | 陀螺积分 + 实时反推 `aFwd`，扫 800 ms 滚动窗，找局部峰 | `aFwd` 三采样局部峰且 `> triggerAccel` → `RELEASED` |
| `RELEASED` | `_release()` 调用 | 计算 `v0/pitch/roll`，构造 `fwd/up`，触发振动，向 `Sim` 发 `launch` | 一次性，进入 `FLIGHT`（由 Sim 持有） |

**关键：进出条件是单向的**——`SWINGING` 一旦进入就没有"超时回退到 `ARMED`"的路径。详见 §16.5 第 9 条。

### 16.2 握持合法性检测（ARMED 阶段）

#### 16.2.1 物理依据

MRD §3.1 规定的标准握姿下，世界重力 `g_w = (0, -g, 0)` 反变换到手机系（§5.2）：

```
g_phone = (g, 0, 0)         ← 重力指向手机 +X (机右)
a_phone = -g_phone = (-g, 0, 0)   ← W3C 加速度计 = 比力 = -gravity（理想静止）
```

所以一个**确实在标准握姿下**且**没在动**的手机，加速度计稳定均值应该是 `(-9.81, 0, 0)`。

#### 16.2.2 三轴阈值（`prototype.html` L352-354）

```js
poseOk = m.x < -poseMain*g          // 主轴下限：m.x < -6.87 (poseMain=0.7)
      && |m.y| < poseTol*g          // 余轴上限：|m.y| < 3.92 (poseTol=0.4)
      && |m.z| < poseTol*g          // 余轴上限：|m.z| < 3.92
```

几何含义：把 `m / g` 看作单位球上的点（理想为 `(-1, 0, 0)`），三个不等式划出一个**锥形容差区**。`poseMain=0.7` → 主轴方向偏差容许 ~45°；`poseTol=0.4` → 每个余轴各容许 ~24°。

#### 16.2.3 200 ms 校准窗口

逻辑（`_feedArmed`）：

```
每帧 push a 进 armSamples
仅当 (s.t - armT0 > 200 ms)：
  m = mean(armSamples)
  if !poseOk:    清空 armSamples，armT0 重置为当前 t —— 滚动重试
  else:          q := Q_CANON, 切到 SWINGING
```

**注意**：窗口内**不会**触发态转换；用户在前 200 ms 即使握得再正也得等满。这避免瞬态噪声拉错均值，但也意味着每次"准备"按下后都有恒定的 200 ms 等待。

### 16.3 投掷检测（SWINGING → RELEASED）

#### 16.3.1 每帧四步（`_feedSwing`）

```
1. q ← Q.integ(q, ω·dt)               // 陀螺积分（一阶）
2. g_phone = q⁻¹ · (0, -g, 0)_w       // 当前姿态下重力在手机系的投影
3. a_lin_phone = a + g_phone           // 减重力得线加速度
4. a_lin_w = q · a_lin_phone           // 旋到世界系
5. aFwd = -a_lin_w.z                   // 世界 forward = -Z 上的投影
```

`aFwd` 是一段时间序列，物理含义：**手机沿出手前进方向的瞬时线加速度**。一次正常挥动会先正后负（加速→甩出后反向减速），形成单峰。

#### 16.3.2 局部峰检测（L394-400）

```js
A=swing[n-3], B=swing[n-2], C=swing[n-1]
if (B.aFwd > triggerAccel
 && B.aFwd >= A.aFwd      // ← 注意是 ≥
 && B.aFwd >  C.aFwd):
   _release(B)
```

**触发延迟**：B 是中间样本，所以 `_release` 必须等到 C 出现，即真实峰值后**约一帧**（16 ms @ 60 Hz）。这意味着触觉反馈和"出手时刻"之间有 ~16 ms 滞后，体感可接受但不严格对齐 MRD §4.2 "脱手瞬间"。

**阈值物理量级**（参考）：
- 把手机轻放到桌上：~5 m/s²
- 一次普通挥手：~15–25 m/s²
- 用力一甩：~30–50+ m/s²
- 当前默认 `triggerAccel = 10`：处在"普通挥手"下沿，**易被慢速擦边动作误触发**（详见 §16.5 第 6 条）。

#### 16.3.3 出手起点回溯（`_release` L420-423）

```js
let startIdx = 0;
for (i = buf.length-2; i >= 0; i--):
   if (buf[i].aFwd < 0.5) { startIdx = i + 1; break; }
```

从 peak 往回找第一个 `aFwd < 0.5 m/s²` 的样本，下一个就是 `startIdx`（"动作开始")。`0.5` 是硬编码常数，未抽进参数面板。

### 16.4 出手初始状态确定（`_release`）

链条三件事：

#### 16.4.1 初速 `v0`（L424-431）

```
v0 = ∫ aFwd dt       (从 startIdx 到 peak，trapezoid 积分)
v0 *= kImpulse        (默认 1.1)
v0 = clamp(v0, vMin=3, vMax=18)
```

物理诠释：积分一段沿 forward 的加速度即沿 forward 的速度增量。**前提**是这段时间内手机的 forward 方向不变——这个前提对短促挥动（<200 ms）勉强成立，但对带翻转的复杂动作不成立（§16.5 第 5 条）。

`kImpulse` 是经验补偿，物理来源可能是：（a）峰值过后还有短暂正向加速度被截断；（b）陀螺积分误差让 forward 投影偏小。两者都是 hack，不是建模。

#### 16.4.2 出手 pitch / roll（L433-442）

```js
sampN = min(5, startIdx + 1)
aAvg  = mean(buf[startIdx-sampN+1 .. startIdx].a)   // 挥动开始前 ~83 ms 的均值
pitch = atan2(-aAvg.y, -aAvg.x)
roll  = atan2( aAvg.z, -aAvg.x)
```

**关键设计取舍**：用挥动**开始前**的稳定重力反推角度，**不用陀螺积分**也**不用挥动中的 a**。
- 优点：挥动期间 `a_lin` 量级远超 `g`，陀螺积分则有漂移，两者都不可靠。挥动前 5 帧时 `a_lin ≈ 0`，`a ≈ -g_phone`，三角函数反解干净。
- 代价：**锁的是出手前姿态**，不是 MRD §3.1 字面要求的"出手瞬间俯仰角"。如果用户在挥动过程中明显翻转手腕（先平后抬），最终入射角与锁定值会差好几度。

#### 16.4.3 拒绝与重建

```js
if |pitch| > 60°:  emit('warn'); arm(); return       // 仅 pitch 检查
fwd = (0, sin p, -cos p)                              // §5.3 两步法
up  = (sin r, cos r·cos p, cos r·sin p)
```

随后 `setState('RELEASED')` + `vibrate(60)` + `emit('release', {v0, pitch, roll, fwd, up})`，`Sim.launch` 接手。

### 16.6 实施摘要（2026-05-10）

修复落地集中在 `prototype.html` 的两处：

**A. `Q.integ` / `Q.conj`（数学层）** — `prototype.html:195-211`

```js
integ: (q, omega, dt) => {
  const hx = omega.x*dt*0.5, hy = omega.y*dt*0.5, hz = omega.z*dt*0.5;
  const theta = Math.hypot(hx, hy, hz);
  const s = theta < 1e-7 ? 1 : Math.sin(theta) / theta;
  const c = theta < 1e-7 ? 1 : Math.cos(theta);
  return Q.norm(Q.mul(q, {x: hx*s, y: hy*s, z: hz*s, w: c}));
}
```

### 16.7 残留事项

- **#7 `kImpulse` 真值标定**：仍保留为经验参数。需要一次"高速摄像 + 加速度同步"的标定实验来固定数值。建议在小规模真机测试时联调。
- **#3 出手姿态精度** 现在依赖陀螺积分的累计精度。`Q.integ` 已升级到指数映射（#13 修复），单帧误差 < 1‰；200ms 内累积漂移在合理 ω 下应 < 2°，但**长时间** SWINGING（接近 swingTimeoutMs）下漂移会显著。如未来发现问题，下一步是引入**互补滤波**：在挥动间歇用 `a` 修正 q 的低频分量，与陀螺高频积分融合。

> **设计哲学迁移**：修前是"挥前重力定一切，挥中只测峰"——稳但失真；修后是"挥前重力初始化 q，挥中陀螺接管，release 用最新 q 取姿态"——更接近 MRD §3.1 字面要求，代价是把陀螺质量纳入了责任面。下一步改进的最大杠杆是**互补滤波**（§16.7 残留 #2）。
